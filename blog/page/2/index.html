Core Graphics工作在graphics context内，graphics context像一块画图，是你绘图操作的目标。在UIiew的drawrect內，ios会自动为你准备好graphics context，你需要做的就是用　UIGraphicsGetCurrentContext()　来获得当前的graphics context。 你在graphics context做的动作是顺序的，新的操作会基于上一次的操作。Core Graphics最常见的操作是重写UIView的drawrect方法，当ios需要重绘View时，它就会调用drawrect。这就表示这个方法会被频繁的调用，尤其在动画/缩放等操作的时候，所以这个方法必须尽量轻，作为开发者，你永远不应该自己调用这个方法，也不该在这里alloc或release对象。

当你需要重绘视图时，你可以用视图的setNeedsDisplay方法，它会告诉ios在合适的时候把这个视图重绘。

当你新建UIView的子类时，你会得到一个全空的graphics context，graphics context有很多的属性，就像其他的IOS对象一样。不像UIVIew，graphics context的属性时状态导向，意思是绘画动作都依赖于当前的属性。

Arc笔记

Mar 23rd, 2012

此篇文字为阅读　http://www.raywenderlich.com/　上关于ARC的指南文章的笔记

ARC(auto reference counting)是苹果在ios5,mac 10.7以后引入的一项内存管理的新技术。目标是要替代手工的内存计数机制，由手工变自动，减少代码量，提高编程效率，减少出错率。它是苹果编译器的一项特性，在代码编译阶段自动插入原来由你手工执行的retain,release等的内存管理相关的代码，编译器会知道如何以及合适插入相关代码。

学习ARC我们要知道的机制是，对象是否释放由指向它的指针来标志，一个对象如果由指向它的指南，它就维持，如果没有，它就被释放！这条规则对实例变量，局部变量，属性由成立.

下面我们来看一个例子

这条语句让firstName成为了指向字符串”Rayn”的指针，”Rayn”还是一个指向它的指针是self.textField.text。如果用户修改了textField.text的值，比如@”Rayman”,textField.text开始保持新的值”Rayman”,这个时候”Rayn”还有一个指向它的指针firstName,所以它在维持在内存中。当firstName也被修改，或者firstName过了它的作用域的时候，也就是不在由指针指向”Rayn”，这个字符串对象就会被销毁。

我们称FirstName和self.textField.text为(strong)强类型因为他们可以保持对象的,默认所有实例变量和局部变量都是强类型。还有一个类型是(weak)弱类型，此种类型的变量不保持对象，加入上面例子中FirstName是弱类型，当textField.title改变后,”Rayn”就会被销毁，因为不再任何指向它的对象。此时FirstName指向的是nil。

弱类型只在一些特定的场合派用场。比如两个对象是父子关系，父对象会用一个强指针来维持子对象，子对象则用弱类型指针来建立对父对像的引用，一旦父对象被销毁，子对象也就销毁了，否则的话子对象也拥有父对象，他们就永远无法销毁了。

有了ARC以后，在内存管理方面我们要思考的主要问题变成了“谁拥有什么？”，请看下面的例子

如果是手工管理维持与释放，上面的代码会报错，因为在NSLog打印obj之前obj已经被销毁了。但是在ARC下会工作，因为[array objectAtIndex:0]对象的值除了数组拥有外obj也拥有它，所以虽然数组里的指针被销毁了，obj还是让它得以保持。

ARC也有一些限制，其中之一是它只对Object-C对象起作用，对Core Foundation或者直接C写的malloc()和free()不起作用，你还是要手工管理。

ARC替我们做了很多工作，但它不是真正意义上的垃圾处理机制，它还是计数管理，所以我们还是要明白的了解其中的运行机制，在特定时候要手动把变量值设为nil以释放，你不能完全依赖它。
